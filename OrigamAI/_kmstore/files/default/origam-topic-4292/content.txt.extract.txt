# How to create a simple API

 This article demonstrates how to create a simple  RESTful API  that makes your data accessible to external applications and services or to internal custom frontend plugins. 
 There are four  different API types  – in ORIGAM called “API Pages” – this text focuses solely on one of these types – the  Data Page, which provides data in XML or JSON format . 
 
   Note: The API in the model is only used to expose the API of the modeled application externally; work with any external APIs is modeled in the business logic section using  sequential workflows . 
 
 
 [!success]- Prerequisites for proceeding 
 To follow along, you must meet the following prerequisites: 
 
  Running database server  
  Running ORIGAM Server  
  Project created in ORIGAM Architect  
  Entity with data created  
 
 
 We will perform the following actions: 
 
 Retrieve an entire list of items 
 Retrieve the details of a specific item from the list 
 Update an existing item 
 Delete an item 
 
 As an example, we will use the simple  task manager from the initial tutorial . We will list all tasks and then work with one of them. 
 
     If you are new to application development or unfamiliar with APIs, you may find the following videos useful: 
 
  API introduction with real-world examples  
  REST API and HTTP methods

 ORIGAM works with both REST and SOAP APIs and currently supports the following  HTTP methods : 
 
 GET 
 POST 
 PUT 
 DELETE 
 
 It supports both  JSON  and  XML  formats; in this example, we will use only JSON. In terms of structure, the data in the API is based on a  data structure . 
 There are two API privacy options:  private  or  public . Any API starting with  api/public  is preset as public, for other URLs the accessibility as public or restricted routes can be declared in the  UserApiOptions  section of the  appsettings.json  file. In other words, anything not located in  api/public/  is considered private by default, unless explicitly set otherwise. 
 In this tutorial, we will create a  public API  — accessible to anyone once published, but limited by the server accessibility, during development typically the localhost. 
 In all cases, we will create a new  Data Page  designed to expose application data:

 When working with APIs, you will usually also need to filter records using  parameters  in the URL — for example, to specify a particular record using its Id, or to select records only within a certain time range (date from — date to). This is done using parameters that need to be defined within the API. 
 You can learn more about other API Page types in the  documentation . 
   Getting ready to create an API 
   Data structure 
 Any API data page requires a  Data Structure  to operate. Before creating the API data page, we’ll first create one. You can name it the same as the main database entity with  _API  suffix. 
 If you plan to include multiple different APIs in your application, it may be clearer to place all the API related data structures into a separate group (subfolder), similar to how it’s done for Lookups.

   Get the details of a specific item from the list 
 To target a specific record, you better create a new API Data Page this time with  {id}  in the name. You can also use the existing API created in the previous step, define a filter based on the Id, and ignore it if the Id is empty — that is, display all existing Ids. However, having a separate API is usually clearer.

    In ORIGAM, the  PUT  command is used for both record creation and updating. The  PATCH  method is currently not used. 
 
 When a PUT request is sent, there are two possible scenarios: 
 A record with the specified Id 
 
 already exists — it will be updated, 
 does not exist — it will be created. 
 
 In both cases, follow these steps: 
 
 Include the Id in the URL, e.g.: 
 PUT https://localhost/api/public/task/5784a17e-15fe-4f1e-b612-6a35055f4c0c  
 Create a separate parameter for the request body in your model:

   If you include only some fields in the request body, the remaining fields will be set to  null  or lost. Therefore, if you intend to update only specific fields (similar to a PATCH request), you must still provide the full record data and modify only the fields you wish to change. 
 
   Delete a record 
 To enable record deletion via the API, set  AllowDELETE  to  True  in your API configuration. 
 When a DELETE request with a valid Id is sent, the corresponding record will be deleted. 
For example, this request: 
 DELETE https://localhost/api/public/task/5784a17e-15fe-4f1e-b612-6a35055f4c0c  
will delete the record with that Id, if it exists. No request body is required.
